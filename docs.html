Overview

lillet is 4613B's custom PROS library: AMCL Monte-Carlo localization (200-particle filter fusing tracking-wheel odometry, IMU heading, and four distance sensors against a ray-cast field model) plus a boomerang-carrot motion controller with a LemLib-compatible API. Everything was designed and tuned in the browser simulator first; the JS and C++ are 1:1 ports of each other.

simulator/index.html      design + tune + validate everything here first
simulator/sim_tests.mjs   automated tests (node sim_tests.mjs) — the behavioural oracle
include/robot_config.hpp  ★ THE tuning file — every parameter lives here
include/ + src/           localization/ (Localizer, ParticleFilter, FieldModel)
                          motion/       (Chassis, PID, ExitCondition, ExpoDriveCurve)
src/main.cpp              wiring + autonomous path (paste simulator export here)

Coordinates: origin at field centre, +x right, +y north, heading clockwise from north. Positions in inches, sensor distances in mm, angles in radians internally / degrees at the API surface.

Getting started

All wiring is three objects; every number they need comes from robot_config.hpp:

#include "robot_config.hpp"

pros::Rotation   fwd_enc(cfg::PORT_FWD_ENCODER);
pros::Imu        imu(cfg::PORT_IMU);
pros::Distance   d_f(cfg::PORT_DIST_FRONT), d_b(cfg::PORT_DIST_BACK),
                 d_l(cfg::PORT_DIST_LEFT),  d_r(cfg::PORT_DIST_RIGHT);
pros::MotorGroup left({cfg::PORT_DRIVE_LF, cfg::PORT_DRIVE_LB});
pros::MotorGroup right({cfg::PORT_DRIVE_RF, cfg::PORT_DRIVE_RB});

lillet::Localizer localizer(fwd_enc, nullptr, imu, d_f, d_b, d_l, d_r,
                                  cfg::makeHwConfig(), cfg::makeMclConfig());
lillet::Chassis chassis(left, right, localizer,
                        cfg::makeDrivetrainSpec(), cfg::makeMotionConfig());

void initialize() {
    imu.reset(true);                    // ~3 s, robot MUST be still
    localizer.start({-48, -48, 0});     // known start pose
}
void autonomous() {
    chassis.setPose(-48, -48, 0);       // reseed at the placed position
    chassis.moveToPose(-24, -24, 45, 3000, {.maxSpeed = 90});  // sim export
}

Autonomous paths: design in the simulator's Auton Plan mode, click Copy LemLib Code, paste into autonomous(). The call shapes are LemLib's, so exports compile unchanged.

Tuning guide — do it in this order

Each step depends on the previous one. All constants live in include/robot_config.hpp (§ numbers below match its sections).

  1. Ports & directions — §1
  2. Tracking wheel radius / TPI, then offset — §2
  3. IMU — §3 (mount check, drift check)
  4. Distance sensor placement + base readings — §4b
  5. Particle filter — §4 (simulator Auto Tune)
  6. Drivetrain geometry — §5 (tape measure)
  7. Motion controller — §6 (symptom-driven, on carpet)
  8. Driver curve — §7 (driver preference)
  9. Live telemetry — §8 (optional — verify you can see live position)

1 · Ports & directions

Set the PORT_* constants. Negative port = reversed device (PROS convention, instead of .set_reversed()). Then verify signs:

  1. Push the robot forward by hand → fwd_enc.get_position() must increase. If not, negate PORT_FWD_ENCODER.
  2. chassis.tank(50, 50) must drive forward. Negate the offending motor ports until it does.

2 · Tracking wheel — radius, TPI, offset, gear ratio

Step A — radius (do first, everything scales off it). Roll the robot exactly 48" forward, read the change in fwd_enc.get_position() = D centidegrees:

FWD_WHEEL_RADIUS_IN = 48 × 36000 / (D × 2π)
example: D = 550 000  →  48×36000/(550000×6.2832) ≈ 0.500"

TPI (ticks per inch) is derived automatically as FWD_TICKS_PER_INCH = 36000 / (2π × radius × gear ratio) — a sanity number, not something you set. Radius 0.5" @ 1:1 → TPI ≈ 11459; radius 1.0" → ≈ 5730.

Gear ratio: FWD_GEAR_RATIO = wheel revolutions per one sensor revolution. Sensor on the wheel axle = 1.0.

Step B — offset. Spin the robot in place exactly 10 rotations against a mark; the encoder accumulates D centidegrees from rotation alone:

FWD_OFFSET_IN = D × radius / 360000     (+ = wheel LEFT of turning centre)
Bad radius shows up as the odometry (purple) trail scaling long/short on straight drives. Bad offset shows up as position error that appears only during turns.

3 · IMU

No constants — but two checks. Calibration: imu.reset(true) runs in initialize(); the robot must be dead still for ~3 s. Drift: leave the robot still 2 minutes and watch the heading on the LCD. More than ~2° → remount (rigid, near the centre of rotation, away from motors) or replace the unit. The whole stack trusts IMU heading absolutely; test how much drift the filter survives with the simulator's IMU drift slider before trusting a marginal IMU.

4 · Distance sensors — placement & base readings

The filter needs to know where each sensor actually is, per sensor, in robot_config.hpp §4b (defaults assume face-centre, perfectly aimed, no bias):

Constant (×4 faces)MeaningHow to measure
DIST_*_FWD_INrobot centre → sensor glass, inches (-1 = use ROBOT_HALF_IN)ruler, centre of robot to the glass
DIST_*_LAT_INposition along the face; + = right of the facing directionruler from the face centreline
DIST_*_YAW_DEGaim error; + = beam points clockwise of straight outusually 0 unless the mount is visibly crooked
DIST_*_BASE_MMbase reading: raw − true distance (recession + zero bias); subtracted from every raw readingprocedure below

Base reading calibration (repeat for each sensor — they differ). No tape measure or 24" spacer needed — opcontrol() has a live readout built in (src/main.cpp, printDistCalibration()):

  1. Drive into driver control and hold Y — the LCD swaps its pose readout for all four sensors' live edge=…in raw=…mm lines.
  2. Push the face you're calibrating flush against a flat wall, squarely.
  3. Read that face's raw= value off the LCD — with BASE_MM still at its default (0) this is the calibration value. Round it and type it into that face's DIST_*_BASE_MM. Typical: 0–40 mm.

Why this works: at the default mount (DIST_*_FWD_IN = -1, i.e. "use ROBOT_HALF_IN"), the sensor glass is assumed to sit exactly at the robot's physical edge — so touching the wall makes the true distance exactly 0, and BASE_MM = raw − true distance collapses to just raw. The readout's edge= value already does that subtraction for you (and folds in any non-default FWD_IN, so it stays edge-relative even for a recessed/proud mount) — once calibrated, touching a wall should show edge=0.00in on every face; anything else means the mount shifted or BASE_MM needs a nudge.

Also in §4: max_range_mm (1900 — the real sensor dies near its 2 m limit) and min_distance_confidence (45 of 63 — raise if the filter twitches near other robots, lower if too few readings survive).

5 · Particle filter (AMCL)

Don't hand-tune this. In the simulator: set Sensor noise σ and Encoder noise to realistic values (20 mm / 0.5%), leave Sensor max range at 2000 mm, click Auto Tune (30 trials), copy the values into makeMclConfig(). Then sanity-check with node sim_tests.mjs.

SymptomTry

6 · Motion controller

On carpet, charged battery, one constant at a time, in this order:

SymptomConstantDirection
Weaves / S-curves along a straight approachMTN_ANG_KPlower
Corners lazily, drifts wide of the carrotMTN_ANG_KPraise
In-place turns overshootMTN_TURN_ACCEL_DEG_S2raise until it just overshoots, back off 20%
Overshoots drive targetsMTN_DECEL_INraise
Crawls in painfully earlyMTN_DECEL_INlower — or set minSpeed per move
Approach arc too wide / too tightMTN_LEADlower / raise (or per-move lead)
Move times out just short of the targetMoveParamslonger timeout or minSpeed ≈ 8–15
Final heading matters (scoring pose)MoveParams::settleHeadingset true on that move
Turns are stall-proof: if the robot is blocked mid-turn the profile re-locks on the residual error and keeps trying until the timeout — no silent idling.

7 · Driver curve

MTN_DRIVE_CURVE_GAIN 0 = linear. 10–20 = soft near centre, full at the rails. DEADBAND kills stick drift (3 is fine); MIN_OUTPUT 8–12 stops static friction eating tiny inputs. Only affects opcontrol feel — never autonomous.

motion/chassis.hpp

Contains prototypes for the lillet::Chassis motion-controller class and its MoveParams per-move override struct. Direct-drive chassis with no internal odometry of its own — every motion call reads Localizer::getPose() as its sole position feedback. moveToPose/moveToPoint/turnToHeading/ turnToPoint port the carrot + kinematic controller validated in the browser simulator, not LemLib's PID boomerang: turn direction/total angle are locked once per motion and driven by an accel/decel velocity ramp instead of live proportional feedback, which is what avoids oscillation on near-180° turns.

Live telemetry — terminal & webpage

The V5 Brain is never wired directly to a computer — only the V5 Controller is, tethered via USB while wirelessly paired to the Brain. PROS already relays a running program's stdout through that link with zero extra setup, so lillet::TelemetryReporter (§8, API reference) just printfs pose telemetry into it.

  1. Terminal (minimum bar): plug the V5 Controller into the PC via USB (wirelessly paired to the Brain, robot on), run pros terminal. Watch TLM x=.. y=.. h=.. c=.. t=.. lines print live during opcontrol() and autonomous().
  2. Webpage (stretch goal): run node simulator/live_server.mjs (spawns pros terminal itself and serves a live view — no npm install, Node built-ins only), then open http://localhost:8973/live.html. The page polls every 150 ms and draws the robot's live position on a field canvas; it shows "NO SIGNAL" if the feed goes stale or the server isn't running.
Ships enabled by default (TELEMETRY_ENABLED = true, §8). It's a fully separate low-priority task from Localizer's — a slow stdout write over the wireless link can only delay a telemetry line, never the pose update that drives the robot — but verify no control-loop hitching on real hardware before trusting that at competition, same as any other new on-robot behaviour.
